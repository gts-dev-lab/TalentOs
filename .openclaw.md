# Instrucciones para asistentes de código (TalentOS)

Este archivo contiene reglas directas para que un asistente de IA (OpenClaw, Cursor, Copilot) pueda trabajar en el proyecto sin contexto previo.

---

## Contexto del proyecto

- **TalentOS** es un LMS (Learning Management System) para formación corporativa.
- Stack: Next.js 15, TypeScript, Dexie (IndexedDB), PostgreSQL opcional, Genkit (IA).
- Arquitectura multi-inquilino: JWT con `tenantId`, middleware, RLS en PostgreSQL.

---

## Reglas conceptuales: no introducir funcionalidades inexistentes

1. **No asumir** que existen sistemas de tickets de negocio (soporte, incidencias, solicitudes de usuarios).
2. En el código **no hay** entidades "tickets" ni lógica de negocio asociada.
3. Las entidades con ciclo de vida (Enrollment, PDI, ComplianceAudit, Course) son **funcionalidades distintas**; no confundirlas con tickets.
4. Antes de implementar cualquier feature, **verificar** que exista en el modelo de dominio o en el plan de migración.

---

## Sistema de tickets: herramienta de organización del desarrollo

Los **tickets** (TT-101 a TT-117) son únicamente una forma de **dividir, organizar y seguir las tareas pendientes de desarrollo**. Documentados en `docs/MIGRATION_PLAN_TICKETS.md`.

- Equivalentes a issues de JIRA/GitHub.
- Representan tareas técnicas concretas.
- NO forman parte del modelo de dominio.
- NO hay entidades ni lógica de tickets en el código.

### Al implementar tareas del plan

1. Identificar el ticket (ej. TT-114) en `docs/MIGRATION_PLAN_TICKETS.md`.
2. Revisar criterios de aceptación y dependencias.
3. Implementar según la arquitectura existente.
4. Actualizar el estado en la tabla de implementación del documento.

---

## Cómo proponer nuevas features sin asumirlas implementadas

1. **No inventar:** Si una funcionalidad no está en `src/lib/types.ts`, `src/lib/db-providers/types.ts` o en la documentación, no existe.
2. **Proponer primero:** Describir la feature, su modelo de datos y flujos antes de implementar.
3. **Independencia:** Si se propone un sistema de tickets de negocio (soporte, incidencias), tratarlo como una funcionalidad nueva e independiente. No reutilizar identificadores TT-XXX.
4. **Validar con documentación:** Revisar `README_ARCHITECTURE.md` y `src/lib/types.ts` para el modelo de dominio actual.

---

## Cómo extender el sistema de tickets del plan

### Añadir o modificar tickets

1. Editar `docs/MIGRATION_PLAN_TICKETS.md`.
2. Mantener formato: ID, Título, Descripción, Dependencias, SP, Criterios de aceptación.
3. Sincronizar estado en `docs/MIGRATION_STATUS.md`.

### Marcar un ticket como hecho

1. Cambiar el estado en la tabla "Estado de implementación" de `MIGRATION_PLAN_TICKETS.md`.
2. Añadir notas con ubicación del código implementado.
3. Si aplica, actualizar `MIGRATION_STATUS.md`.

---

## Reglas para proponer cambios estructurales

1. **Multi-tenant:** Verificar que los cambios respeten el aislamiento por `tenantId` y no permitan fugas entre inquilinos.
2. **DBProvider:** Cualquier cambio en `src/lib/db-providers/types.ts` debe propagarse a Dexie y PostgreSQL.
3. **RLS:** No proponer cambios que debiliten o eliminen políticas RLS sin justificación explícita.
4. **LTI/SCORM:** Evitar cambios que rompan la compatibilidad con estándares LTI 1.3 o SCORM 2004.
5. **Tests:** Incluir o actualizar tests si se modifican middleware, auth, provider o tenant context.

## Antes de implementar

1. Leer `README.md` y `README_DEV.md`.
2. Si la tarea implica un ticket TT-XXX, consultar `docs/MIGRATION_PLAN_TICKETS.md`.
3. Si la tarea implica datos, revisar `src/lib/db-providers/types.ts` y el provider correspondiente.
4. Ejecutar `npm run test` y `npm run test:security` después de los cambios.
